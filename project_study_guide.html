PROJECT OVERVIEW
What is CodeInsight AI?
A full-stack, production-grade AI developer copilot that lets you upload any GitHub repo or ZIP, semantically parse it, embed the code into a vector store, and then chat with the codebase using RAG + LLMs.
🎯 Core Idea You give it a repository. It parses every file, extracts symbols & dependencies, creates vector embeddings, stores them in ChromaDB, and then you can ask natural-language questions like "How does authentication work?" — the system retrieves the right code chunks and streams an LLM answer with file citations.

🔍 Detailed Code Explanation: types.ts

  • export interface Repository: This interface mirrors the database document structure for repository metadata managed by the MongoDB store and passed to the frontend.
    • _id: string; represents the unique identifier auto-generated by MongoDB (ObjectId serialization).
    • name: string; corresponds to the directory or repository name.
    • sourceType: 'github' | 'zip'; is a union type that dictates whether the source was fetched from a remote Git URL or uploaded as a compressed archive.
    • sourceUrl?: string; is an optional property holding the Git clone link (only populated when sourceType is 'github').
    • status: 'queued' | 'parsing' | 'embedding' | 'ready' | 'failed'; acts as a strict finite state machine tracker, letting the client poll and change views depending on the pipeline's progress.
    • lastError?: string; stores the error log if the status transitions to 'failed'.
    • createdAt: string; keeps a ISO timestamp of record insertion.
  • export interface ChatMessage: Dictates the structural contract of conversation logs.
    • role: 'user' | 'assistant' | 'system'; enforces role-based message origin matching standard LLM API payloads (e.g. OpenAI/Anthropic/Gemini specification).
    • content: string; stores the raw textual narrative or markdown response generated.
    • citations?: { filePath: string; startLine: number; endLine: number }[]; is an optional array of references returned during semantic retrieval (RAG) indicating which files, start lines, and end lines were analyzed to construct the answer.

TypeScriptserver/src/routes/auth.ts (pattern)
// JWT-based auth — register creates user, login returns signed token
router.post('/register', async (req, res) => {
  const hash = await bcrypt.hash(req.body.password, 10);
  const user = await User.create({ email, name, passwordHash: hash });
  res.json({ token: jwt.sign({ sub: user._id }, JWT_SECRET) });
});

router.post('/login', async (req, res) => {
  const user = await User.findOne({ email });
  const ok = await bcrypt.compare(password, user.passwordHash);
  if (!ok) return res.status(401).json({ error: 'Invalid credentials' });
  res.json({ token: jwt.sign({ sub: user._id }, JWT_SECRET, { expiresIn: '7d' }) });
});

🔍 Detailed Code Explanation: auth.ts (pattern)

  • POST /register: Securely inserts new user accounts into the database.
    • bcrypt.hash(req.body.password, 10): Asynchronously computes a salted bcrypt hash of the user's plaintext password. The salt work factor (or rounds) is set to 10. Salting makes the hash unique even for identical passwords, neutralizing rainbow table attacks.
    • User.create({ email, name, passwordHash: hash }): Mongoose query that creates and persists a user record with the hashed credentials. Plaintext passwords are never stored in the database.
    • jwt.sign({ sub: user._id }, JWT_SECRET): Generates a stateless JSON Web Token containing the user's document ID as the subject (sub). The signature is encrypted using the JWT_SECRET environment variable to ensure it cannot be spoofed or altered by the client.
  • POST /login: Authenticates returning users and returns an authorization token.
    • User.findOne({ email }): Checks the MongoDB database for a document matching the input email address.
    • bcrypt.compare(password, user.passwordHash): Cryptographically compares the incoming plaintext password against the stored bcrypt hash. This function extracts the salt from the stored hash string and hashes the input password with it to see if the resulting strings match.
    • if (!ok) return res.status(401): Returns an HTTP 401 Unauthorized status code if password validation fails.
    • jwt.sign(..., { expiresIn: '7d' }): If credentials match, generates a JWT configured with a 7-day expiration (expiresIn: '7d'). This enforces session expiration and limits risk if a token is intercepted.

const chatMessageSchema = new Schema({
  role:     { type: String, enum: ["user", "assistant", "system"], required: true },
  content:  { type: String, required: true },
  citations: [{ filePath: String, startLine: Number, endLine: Number }]
}, { timestamps: true });

export const Repository = mongoose.model<RepositoryDocument>("Repository", repositorySchema);

🔍 Detailed Code Explanation: models.ts

  • type RepositoryStatus: A TypeScript string literal union type ensuring type safety when working with repository pipelines. It allows the system to statically check that statuses can only take valid states: "queued", "parsing", "embedding", "ready", or "failed".
  • chatMessageSchema: A sub-document schema nested inside the main Chat schema representing an item in the chat message array.
    • role: Enforces an enumerated string constraint that only accepts "user", "assistant", or "system", matching the standard structural formats expected by modern language model APIs.
    • content: String containing the message content. It has required: true to prevent empty messages from polluting database logs.
    • citations: An array of sub-documents mapping the specific sections of files retrieved via the vector similarity search that the assistant utilized. This allows the frontend to link directly to source files.
    • { timestamps: true }: A Mongoose option that automatically adds and updates createdAt and updatedAt ISO date fields whenever a new message is appended.
  • Model Compilation and Export:
    • Chat: Compiles the Chat schema into a model to execute database actions over the chats collection.
    • Repository: Creates a strongly-typed model leveraging TypeScript interfaces (via the generic parameter <RepositoryDocument>) to interact with the repositories collection.
    • User: Compiles the User schema into a model, leveraging <UserDocument> generic types to allow strong autocomplete on database documents.

Python · Pydanticservices/common/models.py
from pydantic import BaseModel

class CodeSymbol(BaseModel):
    name: str
    kind: Literal["function", "class", "method", "api", "import"]
    filePath: str
    startLine: int
    endLine: int
    signature: str | None = None

class CodeChunk(BaseModel):
    id: str                   # unique chunk ID
    repositoryId: str
    filePath: str
    language: str
    startLine: int
    endLine: int
    content: str              # actual source code
    symbol: CodeSymbol | None = None
    imports: list[str] = []   # import identifiers found in chunk

class DependencyEdge(BaseModel):
    source: str
    target: str
    kind: Literal["imports", "calls", "contains", "api"]

class RepositoryAnalysis(BaseModel):
    repositoryId: str
    files: list[RepositoryFile]
    chunks: list[CodeChunk]
    symbols: list[CodeSymbol]
    dependencies: list[DependencyEdge]
    summary: dict

🔍 Detailed Code Explanation: models.py (Pydantic Models)

  • BaseModel: The base class from Pydantic used to construct strongly-typed data validation models. It enforces strict type parsing, validation checks, and exports data shapes seamlessly into standard formats like JSON or dictionaries.
  • CodeSymbol: Models programmatic declarations extracted from the codebase.
    • name: str; stores the identifier (e.g. "login" or "UserService").
    • kind: Literal[...]; restricts the type of symbol using a Python Literal construct, rejecting values outside of functions, classes, methods, endpoints (api), or module imports.
    • filePath: str, startLine: int, endLine: int; tracks the code symbol coordinates.
    • signature: str | None = None; is an optional field that stores formal signature definitions (such as parameter signatures and returns).
  • CodeChunk: Represents a single, coherent segment of a source file suitable for indexing.
    • id: str; contains a unique chunk hash (usually generated from file coordinates and contents).
    • content: str; stores the raw code block context itself.
    • symbol: CodeSymbol | None = None; links this chunk to a declared symbol if it contains one.
    • imports: list[str] = []; stores module imports found inside the chunk, which helps compile dependencies.
  • DependencyEdge: Defines a directional relationship between symbols or files.
    • source: str, target: str; represents file links or calls.
    • kind: Literal[...]; defines the connection type, such as "imports", "calls" (invocations), "contains" (parent-child blocks), or "api" (endpoint routes).
  • RepositoryAnalysis: The root payload representing a parsed repository.
    • files: list[RepositoryFile]; lists all file nodes.
    • chunks: list[CodeChunk]; holds granular code slices that will be vector-indexed.
    • symbols: list[CodeSymbol]; catalog of all parsed declarations.
    • dependencies: list[DependencyEdge]; stores the complete graph edge configuration representing dependencies.
💡 Why chunks and not whole files? Embedding an entire 2000-line file creates a single noisy vector. A function-level chunk (30–80 lines) creates a precise, semantically meaningful vector — much better retrieval quality.

@app.post("/repositories/{repository_id}/index")
def index_repository(repository_id: str, analysis: RepositoryAnalysis):
    chunks = [chunk.model_dump() for chunk in analysis.chunks]
    texts  = [_embedding_text(chunk) for chunk in chunks]
    embeddings = embed_texts(texts)        # → list[list[float]]
    return upsert_chunks(repository_id, chunks, embeddings)

def _embedding_text(chunk: dict) -> str:
    symbol = chunk.get("symbol") or {}
    return "\n".join([
        f"file: {chunk['filePath']}",
        f"language: {chunk['language']}",
        f"symbol: {symbol.get('name', '')}",
        chunk["content"],
    ])

# Semantic search endpoint also available
@app.post("/repositories/{repository_id}/search")
def semantic_search(repository_id: str, request: SearchRequest):
    embedding = embed_texts([request.query])[0]
    results   = query_chunks(repository_id, embedding, request.limit)
    return {"results": results}

🔍 Detailed Code Explanation: main.py (Embedding Service)

  • @app.post("/repositories/{repository_id}/index"): FastAPI endpoint designated to index parsed chunks of a repository.
    • analysis.chunks: Receives the validation model containing the granular chunks generated by the parser service.
    • chunk.model_dump(): Serializes each Pydantic chunk object back to a standard Python dictionary to enable dictionary operations.
    • _embedding_text(chunk): Prepares the contextual string for each chunk. The string consists of metadata prepended to the actual code block, giving the embedding model semantic cues (e.g. filename, language, symbol name) which significantly improves retrieval accuracy during RAG queries.
    • embed_texts(texts): Calls the shared sentence-transformer configuration to output dense vector arrays (floats).
    • upsert_chunks(repository_id, chunks, embeddings): Sends the list of serialized chunks and their computed embeddings to the ChromaDB client to index the collection under the repository's database identifier.
  • _embedding_text(chunk: dict): Utility helper formatting chunk data.
    • Extracts the nested symbol mapping if it exists, default initializing to an empty dict to avoid KeyError.
    • Formats a newline-separated string template: "file: {filePath}\nlanguage: {language}\nsymbol: {name}\n{content}". Adding the file path and symbol name provides context that enables the embedding model to represent the structural meaning of code (e.g., matching search terms related to file types or module contexts).
  • @app.post("/repositories/{repository_id}/search"): Semantic search test/lookup endpoint.
    • embed_texts([request.query])[0]: Computes a query vector of the user's search text.
    • query_chunks(repository_id, embedding, request.limit): Queries ChromaDB to find the k (limit) closest documents to the query vector, using cosine similarity matching. Returns the matching document payloads and metadata.

The heart of the copilot. Retrieval-Augmented Generation: embed the user query → vector search ChromaDB → build a prompt with retrieved code → stream LLM response with citations.
🧠 What is RAG? RAG = Retrieve, then Augment, then Generate. Instead of asking the LLM to recall code from memory (impossible), we retrieve relevant chunks from ChromaDB and paste them directly into the LLM prompt as context.

Python · FastAPIservices/rag-service/main.py
@app.post("/repositories/{repository_id}/chat/stream")
async def chat_stream(repository_id: str, request: RagRequest):
    embedding = embed_texts([request.query])[0]
    contexts  = query_chunks(repository_id, embedding, 8)
    prompt    = _build_prompt(request.query, contexts)

    async def events():
        async for token in stream_complete(prompt):
            escaped = token.replace("\n", "\\n")
            yield f"data: {escaped}\n\n"    # SSE token event

        citations = [
            { "filePath": item["metadata"].get("filePath"),
              "startLine": item["metadata"].get("startLine"),
              "endLine": item["metadata"].get("endLine") }
            for item in contexts
        ]
        yield f"event: citations\ndata: {json.dumps(citations)}\n\n"
        yield "event: done\ndata: {}\n\n"

    return StreamingResponse(events(), media_type="text/event-stream")

🔍 Detailed Code Explanation: main.py (RAG Service)

  • @app.post("/repositories/{repository_id}/chat/stream"): Asynchronous endpoint that handles incoming chat queries and streams the response back to the client using Server-Sent Events (SSE).
    • embed_texts([request.query])[0]: Encodes the user's natural language question into a vector representing its semantic meaning.
    • query_chunks(repository_id, embedding, 8): Performs similarity query search inside ChromaDB and returns the 8 closest matching code chunks (with contents and metadata) associated with the repository ID.
    • _build_prompt(request.query, contexts): Assembles the system instructions, the retrieved code segments, and the user's question into a single formatted prompt for the LLM.
  • events() Asynchronous Generator:
    • async for token in stream_complete(prompt): Iterates over tokens generated by the LLM in real-time.
    • escaped = token.replace("\n", "\\n"): Escapes newline characters as \n. This is required by the SSE protocol, which uses raw double newlines (\n\n) to delimit messages. Without this escaping, newlines within LLM output would break the stream.
    • yield f"data: {escaped}\n\n": Emits each escaped token inside an SSE-formatted data wrapper.
    • citations = [...]: Builds an array of metadata maps tracking file paths and line ranges for the retrieved code blocks used to formulate the answer.
    • yield f"event: citations\ndata: {json.dumps(citations)}\n\n": Emits a custom SSE event named citations containing the source metadata array in JSON format, allowing the client to show interactive citation cards.
    • yield "event: done\ndata: {}\n\n": Emits a final event signaling that the generation stream has finished.
  • StreamingResponse: Wraps the asynchronous generator in a streaming HTTP response, specifying the content type as text/event-stream. This keeps the connection open while the LLM generates tokens.

🔧
LLM Usage Pattern
Pythonservices/common/llm.py (key function)
async def complete(prompt: str, system: str = "...") -> str:
    chunks = []
    async for chunk in stream_complete(prompt, system):
        chunks.append(chunk)
    return "".join(chunks)   # Full string returned (no streaming)

🔍 Detailed Code Explanation: llm.py (complete function)

  • async def complete(prompt: str, system: str = "...") -> str: An asynchronous utility function that acts as a wrapper around the token-by-token streaming function, turning it into a single blocking response.
    • chunks = []: Initializes an empty list to store incoming text segments (tokens).
    • async for chunk in stream_complete(prompt, system): Iterates asynchronously over the stream of tokens produced by stream_complete (which abstracts calls to Groq/Gemini/OpenRouter).
    • chunks.append(chunk): Appends each incoming text chunk (or word token) to the list as it is returned by the LLM client.
    • return "".join(chunks): Joins all accumulated string components into a single, cohesive response block. This is useful for backend processes that require a full document structure before continuing, such as generating code reviews, summaries, or structured documentation files.

Pythonservices/common/embeddings.py (pattern)
from sentence_transformers import SentenceTransformer

# Model loaded once at startup (singleton pattern)
_model: SentenceTransformer | None = None

def embed_texts(texts: list[str]) -> list[list[float]]:
    global _model
    if _model is None:
        _model = SentenceTransformer("all-MiniLM-L6-v2")  # or CodeBERT
    vectors = _model.encode(texts, convert_to_numpy=True)
    return vectors.tolist()

🔍 Detailed Code Explanation: embeddings.py

  • Singleton Model Loader (_model):
    • _model: SentenceTransformer | None = None: Declares a global module-level variable to cache the initialized model. Loading deep learning models from disk is computationally expensive and memory-heavy.
    • global _model: Explicitly binds the local scope to the module-level variable.
    • if _model is None: _model = SentenceTransformer("all-MiniLM-L6-v2"): Ensures the neural network model is loaded exactly once (when the function is first called by any endpoint) and reused for all subsequent requests. This is the Singleton Pattern, preventing massive memory leaks and CPU overhead.
  • embed_texts(texts: list[str]) -> list[list[float]]:
    • Converts a batch of plain-text code chunks or queries into dense mathematical vector representations.
    • _model.encode(texts, convert_to_numpy=True): Feeds the batch of texts to the transformer. It generates numerical vectors mapping semantic meaning. High-dimensional vector space positioning mirrors textual context (similar concepts end up close together).
    • vectors.tolist(): Converts the high-performance NumPy float arrays into standard Python nested float lists, which can be easily serialized to JSON and stored by vector databases like ChromaDB.
⚠️ How services import common/ Each Docker service sets PYTHONPATH=/services (via Dockerfile). So from common.embeddings import embed_texts resolves to /services/common/embeddings.py. This is why all services build from the same Docker context (context: ./services).

ChromaDB Vector Store
ChromaDB is the vector database. Code chunks are stored as (vector + metadata + content). Semantic search finds the most relevant chunks for any user query in milliseconds.

Key Concepts
ConceptMeaning
CollectionLike a table — one collection per repository, named by repository_id
EmbeddingDense float[] vector representing semantic meaning of a code chunk

Pythonservices/common/vector_store.py (pattern)
import chromadb, os

def _client():
    return chromadb.HttpClient(
        host=os.getenv("CHROMA_HOST", "localhost"),
        port=int(os.getenv("CHROMA_PORT", "8000"))
    )

def upsert_chunks(repo_id: str, chunks: list[dict], embeddings: list[list[float]]):
    col = _client().get_or_create_collection(repo_id)
    col.upsert(
        ids=[c["id"] for c in chunks],
        embeddings=embeddings,
        documents=[c["content"] for c in chunks],
        metadatas=[{"filePath": c["filePath"], "startLine": c["startLine"],
                    "endLine": c["endLine"]} for c in chunks]
    )

def query_chunks(repo_id: str, embedding: list[float], n: int) -> list[dict]:
    col = _client().get_collection(repo_id)
    results = col.query(query_embeddings=[embedding], n_results=n)
    return [
        {"content": doc, "metadata": meta}
        for doc, meta in zip(results["documents"][0], results["metadatas"][0])
    ]

🔍 Detailed Code Explanation: vector_store.py

  • _client(): Connection helper.
    • chromadb.HttpClient(...): Lazy-initializes and returns a client pointing to the ChromaDB daemon. ChromaDB is run as a separate service inside Docker container space.
    • os.getenv("CHROMA_HOST", "localhost"): Resolves host routing. When running locally, it defaults to localhost; in Docker Compose, it resolves to the container host name ("chromadb").
  • upsert_chunks(repo_id, chunks, embeddings): Idempotent write operation.
    • col = _client().get_or_create_collection(repo_id): Fetches the vector space index partition designated for this repository, creating it if it is the first time the repo is indexed. Dividing documents into per-repository collections isolates queries, meaning searches are fast and never leak chunks across different repositories.
    • col.upsert(...): Idempotently inserts or updates vectors and documents. Using upsert prevents index duplication if the user triggers repository re-indexing.
      • ids: Unique hash keys identifying each chunk block.
      • embeddings: Numerical vector float matrices generated by the embedding model.
      • documents: The source code lines stored as plain text. This content is retrieved and injected directly into LLM prompt contexts at query time.
      • metadatas: A collection of key-value pairs matching each vector (e.g. filePath, startLine, endLine). These tags can be queried directly, and serve as citations.
  • query_chunks(repo_id, embedding, n): Similarity search query.
    • col.query(query_embeddings=[embedding], n_results=n): Executes cosine similarity matching inside ChromaDB. It compares the dense vector representation of the user's question against all indexed chunks in the repository collection, returning the n closest snippets.
    • zip(results["documents"][0], results["metadatas"][0]): Combines the returned file text and metadata dictionary, producing a list of dictionaries with structure: {"content": doc, "metadata": meta}.

async def stream_complete(prompt: str, system: str = "...") -> AsyncIterator[str]:
    provider = os.getenv("LLM_PROVIDER", "mock").lower()

    if provider == "groq" and os.getenv("GROQ_API_KEY"):
        async for token in _stream_openai_compatible(
            "https://api.groq.com/openai/v1/chat/completions",
            os.environ["GROQ_API_KEY"],
            os.getenv("GROQ_MODEL", "llama-3.3-70b-versatile"),
            prompt, system
        ): yield token; return

    if provider == "gemini" and os.getenv("GEMINI_API_KEY"):
        yield await _complete_gemini(prompt, system); return

    yield _offline_answer(prompt)  # fallback mock

🔍 Detailed Code Explanation: llm.py (Provider Routing)

  • async def stream_complete(prompt: str, system: str = "...") -> AsyncIterator[str]: Asynchronous generator function that yields text tokens from the LLM, abstracting away different model providers.
    • provider = os.getenv("LLM_PROVIDER", "mock").lower(): Reads the configured AI backend from environment variables, defaulting to "mock" for offline execution.
    • Groq Integration:
      • Checks if provider == "groq" and that the corresponding API key exists.
      • _stream_openai_compatible(...): Connects to Groq's OpenAI-compatible endpoint. It forwards authorization headers, specifies the LLaMA model, and streams generated text chunks using standard Server-Sent Events (SSE).
      • yield token; return: Yields tokens sequentially to the caller. The generator terminates immediately on completion (return) to prevent fallbacks.
    • Gemini Integration:
      • Checks if provider == "gemini" and that the corresponding Google API key is configured.
      • yield await _complete_gemini(prompt, system): Calls the Gemini REST endpoint. Since Gemini's standard REST endpoints do not stream using OpenAI's SSE chunk format, it returns the complete generated response in one block.
    • Mock Fallback:
      • If the environment variable is unset, or set to "mock", the generator executes _offline_answer(prompt). This helper returns a static message followed by a truncation of the compiled prompt, enabling local testing and verification without network requests or API costs.
⚠️ Groq vs Gemini streaming difference Groq and OpenRouter use OpenAI-compatible SSE streaming (data: [DONE] pattern). Gemini uses a different REST API and doesn't natively stream the same way — so the Gemini path uses complete() not stream_complete(), returning the full response at once.

// Applied to all routes except /api/auth/*
const authMiddleware = (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];  // "Bearer <token>"
  if (!token) return res.status(401).json({ error: 'No token' });
  const payload = jwt.verify(token, JWT_SECRET);
  req.userId = payload.sub;   // attach userId to request
  next();
};

🔍 Detailed Code Explanation: authMiddleware

  • const authMiddleware = (req, res, next) => { ... }: Express custom middleware function used to protect routes from unauthorized access.
    • It intercept HTTP requests, verifying user credentials prior to routing the request payload to CRUD or microservice proxy handlers.
    • req.headers.authorization?.split(' ')[1]: Parses the HTTP Authorization header. The safe-navigation operator (?.) avoids throwing runtime errors if the header is missing. It splits the string by spaces to isolate the token from the "Bearer <token>" format schema.
    • if (!token) return res.status(401).json({ error: 'No token' }): If the header is missing or does not contain a token, the middleware halts request processing and returns an HTTP 401 Unauthorized status, protecting downstream resources.
    • jwt.verify(token, JWT_SECRET): Decodes and cryptographically validates the token signature using the server secret key. If the token has been altered, or has expired, this function throws an error, causing the middleware chain to break (usually caught by a global error handler or returning 403 Forbidden).
    • req.userId = payload.sub;: Extracts the token's subject (sub) property—which is the user's unique database object ID—and binds it to the request object as req.userId. This allows downstream handlers to know which user is making the request and enforce proper ownership checks.
    • next();: Invokes the callback function to pass control to the next handler or middleware in the Express routing stack.

INTERVIEW PREP
Q & A
Click each question to reveal the answer. Test yourself before reading.
What is RAG and why is it used here instead of fine-tuning?›
RAG (Retrieval-Augmented Generation) retrieves relevant documents at query time and injects them into the LLM prompt as context. Fine-tuning bakes knowledge into model weights — you'd need to fine-tune again every time a new repository is uploaded. RAG is dynamic, works with any repo immediately, and the LLM doesn't need to "remember" code it's never seen. It also provides verifiable citations.
Why are there separate services for parsing, embedding, and RAG instead of one Python monolith?›
Microservices allow independent scaling (embedding is CPU/GPU intensive — scale it separately), independent deployment (update RAG without touching the parser), and clear boundaries of concern. Each service can be restarted or debugged in isolation. The common/ library prevents code duplication while keeping services independent.
Why does the parser extract "chunks" instead of embedding whole files?›
Embedding a 2000-line file creates one noisy vector that captures everything loosely. A function-level chunk (30–80 lines) creates a precise, semantically focused vector. When a user asks "how does login work?", the similarity search returns the exact login() function chunk — not the whole auth.ts file. Chunk-level embeddings dramatically improve retrieval precision.
How does streaming work technically (SSE)?›
Server-Sent Events (SSE) is a one-directional HTTP stream. The server keeps the connection open and sends data: <content>\n\n messages. FastAPI returns a StreamingResponse. The Node gateway proxies this stream. The React client reads it with fetch() + ReadableStream. Each token from the LLM becomes one SSE message, creating the word-by-word streaming effect.
What is ChromaDB and why not use a regular SQL database for vectors?›
ChromaDB is a purpose-built vector database that stores dense float[] embeddings and enables fast approximate nearest-neighbor (ANN) search using cosine similarity. A regular SQL database can store float arrays, but querying "find the 8 most similar vectors to this query vector" across 100k rows would be extremely slow without specialized indexing (like HNSW, which ChromaDB uses internally). ChromaDB is optimized for exactly this semantic similarity use case.
Why does the embedding text include file, language, and symbol name — not just the code?›
The embedding model (sentence-transformers) creates a vector that represents the semantics of the input text. By prepending contextual metadata — "file: src/auth/login.ts, language: typescript, symbol: loginUser" — the embedding captures not just what the code does, but where it lives and what it's called. This means a query like "the TypeScript login function" will more likely retrieve the right chunk vs just embedding raw code.
What happens if the repository has no LLM provider configured?›
The _offline_answer() function in common/llm.py is called. It returns a static string explaining that the LLM is not configured, followed by the first 3000 characters of the constructed prompt (which includes the retrieved code context). The system still works — you can see which code chunks were retrieved — but there's no LLM-generated answer.
How does the repository status flow through the system?›
Status is stored in MongoDB as a simple string field. The Node gateway drives the state machine: on upload → "queued" → calls parser → "parsing" → receives analysis → calls embedder → "embedding" → embedding done → "ready". If any step throws an error, the gateway catches it, sets status to "failed" and stores the error message in lastError. The frontend polls the status endpoint every few seconds until it sees "ready" or "failed".
Why does the Node.js gateway exist — why not call Python services directly from the frontend?›
Security and separation of concerns. The gateway (1) handles authentication — Python services don't need to implement JWT, (2) enforces access control — users can only access their own repos, (3) manages MongoDB — Python services stay stateless, (4) orchestrates the multi-step parse → embed pipeline, (5) is a single CORS-enabled entry point for the browser — the browser never directly reaches Python services.
