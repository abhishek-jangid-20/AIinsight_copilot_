RAG & AI Pipeline — 12 Questions
CATEGORY 1 · 12 QUESTIONS
RAG & AI Pipeline
Questions about the Retrieval-Augmented Generation architecture, retrieval quality, prompt construction, and AI pipeline design decisions.
Q1
What is RAG and why did you use it instead of fine-tuning an LLM?
EASY ›

RAG (Retrieval-Augmented Generation) is an architecture that retrieves relevant documents from an external knowledge store at query time and injects them into the LLM's prompt as context, enabling the model to answer questions about private, up-to-date, or domain-specific knowledge it wasn't trained on.

I chose RAG over fine-tuning for several reasons: Fine-tuning requires expensive GPU compute, retraining every time the codebase changes, and risks "forgetting" general reasoning capabilities (catastrophic forgetting). RAG is updateable in real-time — indexing a new repository just runs the embedding pipeline. Fine-tuning also can't cite specific file and line numbers; RAG returns the actual chunks used, enabling source citations in the UI.

💡 Key phrase Say: "RAG gave me grounded, verifiable answers with citations, while fine-tuning would require retraining every time the codebase changed."
Q2
Walk me through the complete RAG pipeline in your project from user question to displayed answer.
MEDIUM›

1. User types a question in the React chat panel. The frontend sends a POST to /api/repositories/:id/chat on the Node gateway.

2. Gateway proxies to the RAG service (http://rag-service:8103).

3. Query embedding: The RAG service calls embed_texts([query]) to convert the question to a 384-dim vector using all-MiniLM-L6-v2.

4. Vector retrieval: query_chunks(repo_id, embedding, 8) queries ChromaDB's HNSW index, returning the 8 most semantically similar code chunks.

5. Prompt construction: _build_prompt() injects the 8 chunks as numbered context blocks into the LLM prompt.

6. LLM streaming: stream_complete(prompt) calls Groq/Gemini/OpenRouter and yields tokens via SSE.

7. Client rendering: React appends each token to the message as it arrives. After event: done, citations (file paths + line ranges) are displayed.

Q3
How does vector similarity search work and why is cosine distance used?
MEDIUM›

Vector similarity search finds the vectors in a database closest to a query vector. Cosine distance measures the angle between two vectors: cosine_similarity = (A · B) / (|A| × |B|). Two vectors pointing in the same direction have cosine similarity 1 (identical meaning); perpendicular vectors score 0; opposite score -1.

When we normalize embeddings (divide by their magnitude), all vectors have length 1. This means cosine similarity equals dot product for normalized vectors — a much cheaper operation. ChromaDB uses HNSW (Hierarchical Navigable Small World) graphs to find approximate nearest neighbors in O(log N) time rather than brute-force O(N).

I use normalize_embeddings=True in embed_texts() specifically to enable this optimization.

Q4
Why retrieve 8 chunks? How did you decide the optimal retrieval count?
MEDIUM›

8 is a heuristic tradeoff. Each code chunk is ~120 lines ≈ 1,500-2,000 tokens. With 8 chunks, the context portion of the prompt is ~12,000-16,000 tokens — well within Groq Llama-3's 32K context window while leaving room for the question and system prompt.

Retrieving fewer chunks risks missing relevant code. Retrieving more dilutes the context with less relevant chunks, which can confuse the LLM and increase cost. For production, I'd tune this empirically using precision@k metrics.

💡 Follow-up answer "In production I'd implement retrieval evaluation using a test set of questions with known relevant chunks, measuring Precision@8, Recall@8, and Mean Reciprocal Rank (MRR)."
Q5
How does your prompt prevent the LLM from hallucinating answers?
MEDIUM›

The prompt explicitly instructs: "Answer the developer's repository question using only the retrieved code context. Be specific, cite file paths and line ranges, and call out uncertainty."

Three mechanisms prevent hallucination: (1) The word "only" grounds the model in provided context. (2) Asking for file:line citations means the model must point to real chunks — it can't fabricate a fictional file path. (3) "Call out uncertainty" instructs the model to say "I don't see this in the context" rather than guessing, which is a valid and honest response for a code assistant.

Q6
What is the difference between RAG and traditional keyword search?
EASY›

Keyword search (BM25/TF-IDF) matches documents containing the exact query words. It's fast but semantically shallow — "extract Python functions" won't find code with ast.FunctionDef if those exact words don't appear.

Vector search (RAG) embeds both query and documents into semantic space. Semantically related text clusters together regardless of word choice. "Extract Python functions" will retrieve ast.FunctionDef chunks because both concepts live near each other in embedding space.

For an ideal system, hybrid search combines both: keyword search for exact identifiers (class names, function names) and vector search for semantic queries.

Q7
How do citations work in your system? How are they associated with the LLM answer?
MEDIUM›

After the last LLM token streams, the RAG service sends a custom SSE event: event: citations\ndata: [...] containing the filePath, startLine, and endLine for each of the 8 retrieved chunks (not just the ones the LLM referenced).

The client's SSE listener handles the citations event type separately: it appends the citations array to the last assistant message object, which the UI renders as clickable source links below the answer.

⚠️ Limitation The system shows all retrieved chunks as citations, not just those the LLM actually cited. A more precise system would parse the LLM's [1], [2] references and map them back to specific chunks.
Q8
What is the offline/mock mode and why is it important?
EASY›

When LLM_PROVIDER is not set or no API key is available, llm.py falls back to _offline_answer(), which returns the retrieved context itself as the "answer" — essentially showing the user the raw code chunks that would have been sent to the LLM.

This is crucial for development: it lets me test the entire parsing, embedding, and retrieval pipeline without any API keys or internet connection. I can verify that the right chunks are being retrieved before integrating the LLM. It also means the project can be run and demonstrated without any paid API access.

Q9
How would you improve retrieval quality? What are the limitations of the current approach?
HARD›

Current limitations: (1) Pure vector search misses exact identifier matches. (2) Chunk boundaries can split context awkwardly. (3) No query expansion or reformulation. (4) Retrieved chunks aren't re-ranked for quality.

Improvements I'd make:

  • Hybrid search: Combine BM25 (keyword) with vector search via Reciprocal Rank Fusion
  • Cross-encoder re-ranking: Use a model like ms-marco-MiniLM to score all 20 candidates and return the best 8
  • HyDE (Hypothetical Document Embeddings): Generate a hypothetical code answer, embed it, then search — often retrieves better results
  • Parent chunk retrieval: Store small chunks for precise retrieval but return their parent (larger context) to the LLM
  • Query decomposition: For complex questions, decompose into sub-queries and merge results
Q10
Why does the embedding service prepend file path and language to each chunk before embedding?
MEDIUM›

Prepending "file: services/rag-service/main.py\nlanguage: Python\nsymbol: chat_stream\n" before the source code enriches the semantic signal the model encodes.

When a user queries "Python streaming endpoint", the query embedding will land near vectors that include the word "Python" and "stream" in their metadata prefix — even if the code itself doesn't contain those exact words. It's metadata-augmented semantic indexing: you're teaching the model to associate file paths and languages with specific code patterns.

Q11
What is the OpenAI-compatible API format and how does it work in your LLM abstraction?
MEDIUM›

OpenAI's Chat Completions API has become an industry standard. The request format is: {"model": "...", "stream": true, "messages": [{"role": "system", "content": "..."}, {"role": "user", "content": "..."}]}. The streaming response sends data: {"choices":[{"delta":{"content":"token"}}]} lines.

Groq and OpenRouter both implement this exact format. My _stream_openai_compatible() function works for both — the only differences are the URL and API key. This is the Strategy Pattern: one implementation, multiple providers, swapped via config.

Gemini uses a different format (generativelanguage.googleapis.com REST API), so it has its own _complete_gemini() implementation that makes a blocking call and yields the full response as one token.

Q12
How would you evaluate the quality of your RAG system?
HARD›

Retrieval metrics: Create a golden dataset of questions + known relevant chunks. Measure Precision@k (what fraction of retrieved chunks are relevant), Recall@k (what fraction of relevant chunks were retrieved), and MRR (Mean Reciprocal Rank).

Generation metrics: Use frameworks like RAGAS (RAG Assessment): Faithfulness (does the answer contradict the context?), Answer Relevancy (does it answer the question?), Context Precision, Context Recall.

End-to-end metrics: Human evaluation on a test set, A/B testing between retrieval strategies, tracking citation accuracy (do cited line numbers actually contain the referenced code?).

CATEGORY 2 · 10 QUESTIONS
Embeddings & Vectors
Deep-dive questions on sentence transformers, vector dimensions, ChromaDB indexing, and embedding quality.
Q13
What is a sentence transformer and how does it produce embeddings?
MEDIUM›

A sentence transformer is a BERT-based neural network fine-tuned with a Siamese/triplet network objective: similar sentence pairs are trained to produce nearby embeddings, dissimilar pairs produce distant embeddings (contrastive learning).

The process: tokenize input text → pass through transformer layers (6 layers for MiniLM) → mean pool the token embeddings to get a fixed-size sentence vector → optionally L2-normalize. The result is a dense vector where geometric proximity corresponds to semantic similarity.

all-MiniLM-L6-v2 is a 22M parameter distilled model: fast on CPU (~5ms per text), 384-dim output, trained on 1B+ sentence pairs from web + scientific text.

Q14
Why 384 dimensions? What's the tradeoff between embedding dimension size?
HARD›

384 is the native output dimension of all-MiniLM-L6-v2. The tradeoff: higher dimensions (768, 1536) can encode more fine-grained semantic distinctions but require more storage (2× the bytes per vector), slower ANN indexing, and are prone to the "curse of dimensionality" where distances become less meaningful in very high-D space. Lower dimensions are faster and cheaper but lose nuance.

For code search, 384 dimensions is a practical sweet spot — sufficient for distinguishing between different code concepts while remaining fast enough for real-time retrieval. Production systems like OpenAI's text-embedding-3-small offer 1536 dims but allow truncation to 256 for speed.

Q15
How does the @lru_cache singleton pattern work for the embedding model?
MEDIUM›

@lru_cache(maxsize=1) is Python's built-in memoization decorator. The first call to _load_sentence_transformer() executes the function body (loads the model) and caches the result. All subsequent calls return the cached object without re-executing — effectively a singleton.

maxsize=1 means the cache holds at most 1 entry. Since the function takes no arguments (the model name is read from env inside), the cache key is always the same empty tuple (). The model is loaded once at first request and persists for the lifetime of the process. This avoids loading 80MB from disk on every request.

Q16
Explain the hash embedding fallback. Is it really useful?
MEDIUM›

The _hash_embedding() function creates a deterministic 384-dim vector by: tokenizing text by whitespace → SHA-256 hashing each token → using the first 4 bytes as a vector index → incrementing vector[index] += 1.0 → L2-normalizing the result.

It's a bag-of-words approximation: same tokens → same vector positions incremented. It's NOT semantic (synonyms produce completely different vectors), but it's deterministic — the same text always produces the same vector, so the database is consistent. For offline development, you can index and retrieve documents, but query results will be essentially random. The real value is making the system runnable and demonstrable with zero ML dependencies.

Q17
What is HNSW and how does ChromaDB use it for ANN search?
HARD›

HNSW (Hierarchical Navigable Small World) is a graph-based Approximate Nearest Neighbor index. It builds a multi-layer graph where each node connects to its nearest neighbors at different resolutions. Higher layers have sparser, long-range connections (for coarse navigation); lower layers have dense, local connections (for fine-grained search).

Query time: Start at the entry point in the top layer, greedily navigate to the nearest neighbor in that layer, descend, repeat until the bottom layer. This finds approximate nearest neighbors in O(log N) time vs O(N×d) for brute force. ChromaDB uses HNSW via its hnswlib backend. The tradeoff: ANN means results are "good enough" but may miss the true nearest neighbor — acceptable for semantic search where approximate results are often equally useful.

Q18
Why does each repository get its own ChromaDB collection?
EASY›

Collection isolation provides: (1) Clean deletion — deleting a repository deletes its entire collection without touching others. (2) No cross-repo contamination — queries return chunks only from the target repository, not from other users' repos. (3) Independent scaling — collections can be sharded independently. (4) Tenant isolation — in multi-user scenarios, users can't accidentally retrieve code from other users' repositories.

The alternative (one collection + metadata filter) would require querying all vectors then filtering — slower and harder to delete.

Q19
What is the difference between all-MiniLM-L6-v2 and CodeBERT? When would you use each?
MEDIUM›

all-MiniLM-L6-v2: 22M params, 384-dim, trained on general text including technical content. Fast on CPU. Best for natural-language-to-code search (user asks in English, finds relevant code).

CodeBERT: 125M params, 768-dim, pre-trained on CodeSearchNet corpus (6 programming languages: Python, Java, JS, PHP, Ruby, Go). Best for code-to-code similarity and cross-language code search, but slower and larger memory footprint.

For CodeInsight's primary use case (NL questions → code answers), MiniLM's speed advantage with acceptable quality makes it the better default. CodeBERT would shine if the use case were "find me code similar to this snippet."

Q20
How is the ChromaDB upsert operation idempotent?
MEDIUM›

Idempotent means calling the operation multiple times has the same result as calling it once. ChromaDB's upsert() (unlike add()) will insert a document if its ID doesn't exist, or update it if it does — never creating duplicates.

The chunk IDs are SHA-1 hashes of {file_path}:{start}:{end}:{content[:80]}. If you re-index the same file, the same hash is produced → the upsert updates the existing vector → no duplicates. This means you can safely re-run the embedding pipeline after code changes — modified functions get updated vectors, unchanged functions are a no-op.

Q21
What does normalize_embeddings=True do mathematically?
HARD›

L2 normalization divides each vector by its Euclidean norm: v_normalized = v / ||v||₂. This projects all vectors onto the surface of a unit hypersphere (all vectors have length 1).

The mathematical consequence: for unit vectors, cosine_similarity(A, B) = A · B / (|A| × |B|) = A · B (since |A| = |B| = 1). So cosine similarity becomes a simple dot product — one multiply-accumulate operation per dimension instead of two (dot product + two norm computations). This makes distance computation significantly faster, enabling ChromaDB's HNSW to use inner product metric instead of the more expensive cosine metric.

Q22
How would you handle embedding model updates without re-indexing everything?
HARD›

Changing the embedding model invalidates all stored vectors (a 384-dim MiniLM vector is incompatible with a 768-dim CodeBERT vector — they can't be compared). This is a major operational challenge called embedding drift.

Strategies: (1) Version collections: store model name/version in collection metadata, query only same-version collections. (2) Dual indexing: run old and new models in parallel during migration, gradually deprecate old. (3) Background re-indexing: on model update, queue all repositories for re-embedding without downtime. (4) Pinning: pin the model version in config and only update with a planned migration job. In CodeInsight, I'd store the model name alongside each collection and refuse to query if the current model differs from the stored model name.

CATEGORY 3 · 10 QUESTIONS
System Architecture
Questions on microservices design, the API gateway pattern, inter-service communication, and architectural tradeoffs.
Q23
Why use an API Gateway pattern instead of having the client call Python services directly?
MEDIUM›

The API Gateway (Node/Express) provides: (1) Single entry point — client only needs one URL, not 5 different Python service URLs. (2) Authentication — JWT verification in one place instead of in every Python service. (3) Authorization — ownership checks (this repo belongs to this user) happen before proxying. (4) Request routing — the gateway decides which service to call based on the route. (5) Protocol translation — the gateway handles file uploads (multipart/form-data) from the client and translates them to JSON payloads for Python services. (6) CORS — handle cross-origin in one place.

Q24
How does the repository processing pipeline work as a state machine?
MEDIUM›

The repository's status field acts as a finite state machine: queued → parsing → embedding → ready (or failed at any step). Each transition is a MongoDB update: when the gateway starts calling the parser, it sets status: "parsing"; when the parser responds successfully, it sets status: "embedding"; after ChromaDB indexing completes, it sets status: "ready".

The React client polls GET /api/repositories/:id every few seconds and updates the UI based on the status field. When status reaches ready, polling stops and the chat interface unlocks. If any step throws, status is set to failed with the error in lastError.

Q25
What is the shared common module pattern and why is it better than duplicating code?
EASY›

The services/common/ directory contains embeddings.py, llm.py, vector_store.py, and models.py. In docker-compose.yml, this directory is bind-mounted into every Python service container, and each Dockerfile adds it to PYTHONPATH. Any service can then do from common.embeddings import embed_texts.

Without this pattern: every service would copy-paste the embedding code, and fixing a bug in embed_texts would require changing 5 files. The shared module means one fix propagates everywhere — this is the DRY (Don't Repeat Yourself) principle applied at the service level.

Q26
What are the tradeoffs of synchronous vs asynchronous repository processing?
HARD›

Current approach (synchronous): The gateway makes a blocking HTTP POST to the parser, waits for the response (which can take 30-120 seconds for large repos), then calls the embedding service, waits again, then updates MongoDB. During this time, the HTTP connection is held open.

Problem: HTTP connections have timeouts. Large repos can exceed Nginx/load-balancer timeouts (typically 60-120s). The gateway process is blocked, consuming a thread/connection slot.

Better approach (async with queue): The gateway immediately returns 202 Accepted with a job ID. A background worker (Celery + Redis, or BullMQ) picks up the job and runs parse→embed. The client polls for status. This is much more scalable and production-appropriate.

Q27
How does Docker Compose networking work between services?
EASY›

Docker Compose creates a default bridge network for all services in the same docker-compose.yml. Each service is assigned a DNS name equal to its service name in the YAML. Services communicate using these DNS names as hostnames: http://repository-parser:8101, http://rag-service:8103.

Docker's embedded DNS resolver maps service names to the container's current IP. This is dynamic — if a container restarts and gets a new IP, the DNS name still works. localhost inside a container refers only to that container, NOT to other containers or the host machine.

Q28
Why was Node.js chosen for the gateway instead of Python FastAPI?
MEDIUM›

Node.js + TypeScript excels at the gateway's specific workload: (1) Non-blocking I/O — Node handles many concurrent HTTP connections efficiently, ideal for an API gateway that mostly proxies requests. (2) Mongoose — MongoDB's most mature and feature-rich ODM is in the Node ecosystem. (3) TypeScript — shared type definitions with the React client means the same Repository interface is used in both places. (4) Familiarity — the npm ecosystem has excellent libraries for JWT (jsonwebtoken), bcrypt, file uploads (multer), and CORS.

A Python FastAPI gateway would work too, but would lose TypeScript type sharing with the frontend and has a less mature MongoDB ODM.

Q29
How does the architecture handle failure — what happens if the embedding service crashes during indexing?
HARD›

Currently: the gateway's try/catch around the embedding service HTTP call catches the error and sets status: "failed" + lastError: errorMessage in MongoDB. The client displays the error. No automatic retry exists.

Production improvement: Implement idempotent retry with exponential backoff. Since the embedding step uses upsert() (not add()), re-running it on the same data is safe — already-indexed chunks are updated, not duplicated. A job queue would handle retries automatically, track attempt counts, and dead-letter queue permanently failing jobs for human inspection.

Q30
What is the Vite dev proxy and why is it needed?
EASY›

In development, the React app runs on localhost:5173 (Vite) and the Node gateway on localhost:8080. Direct browser requests from :5173 to :8080 would be blocked by CORS (different origins).

The Vite proxy (vite.config.ts: "/api" → "http://localhost:8080") makes the browser believe all requests go to the same origin. Vite's dev server intercepts any request to /api/* and forwards it to port 8080 — so from the browser's perspective, it's talking to localhost:5173/api (no CORS). This is purely a development convenience; in production (Docker), React is served by the same gateway or a CDN.

Q31
How would you scale this system to handle 10,000 concurrent users?
HARD›

Horizontal scaling per bottleneck:

  • RAG/Embedding services: Stateless — scale horizontally with Kubernetes pods behind a load balancer. Model loaded per pod instance.
  • Parser service: CPU-intensive — scale with a job queue (Celery workers), auto-scale workers based on queue depth.
  • ChromaDB: Scale with ChromaDB's distributed mode or migrate to Pinecone/Weaviate for managed vector search.
  • MongoDB: Replica sets for read scaling, sharding for write scaling.
  • Node gateway: Stateless — scale horizontally, JWT verification needs no shared state.
  • LLM: Rate limit per user, queue long-running requests, cache common query embeddings in Redis.
Q32
What is the Repository Analysis Pydantic model and why is it the central data contract?
MEDIUM›

RepositoryAnalysis is the Pydantic model returned by the parser service and consumed by the embedding, documentation, and analysis services. It contains: repositoryId, files (list of RepositoryFile), chunks (list of CodeChunk), symbols (list of CodeSymbol), dependencies (list of DependencyEdge), summary (dict).

It's the central contract because Pydantic validates the data on both sides (parser output, service input) — any mismatch fails with a clear validation error, not a runtime KeyError. FastAPI automatically serializes/deserializes Pydantic models to/from JSON, making it the API contract between services. It's stored in MongoDB (as Mixed type) so it can be served to the client for architecture visualization without re-calling the parser.

CATEGORY 4 · 10 QUESTIONS
Code Parsing & AST
Questions on repository parsing, abstract syntax trees, code chunking, symbol extraction, and dependency graph construction.
Q33
What is an AST and why is it better than regex for Python symbol extraction?
MEDIUM›

An Abstract Syntax Tree is a tree representation of code's grammatical structure. Python's ast.parse(source) produces a tree of nodes like Module → FunctionDef → arguments → body. Each node has lineno and end_lineno attributes giving exact line coordinates.

AST is better than regex because: (1) it's grammar-aware — handles decorators, nested functions, async functions, lambda expressions that break regex. (2) it provides exact end_lineno (where the function ends) which regex can't reliably determine. (3) it's language-validated — if ast.parse() succeeds, the code is syntactically valid Python. Regex can match false positives in strings/comments.

Q34
How does the two-pass chunking strategy work?
MEDIUM›

Pass 1 (Symbol-level): For each detected symbol (function, class), create a chunk covering exactly that symbol's line range. Symbol-level chunks have rich semantic signal — the embedding captures the full function in context.

Pass 2 (Sliding window): For files with no symbols (config files, plain scripts) or files longer than 200 lines, create 120-line overlapping windows covering every line. This ensures 100% file coverage — no code is left unindexed.

Deduplication: Both passes may generate chunks with overlapping line ranges. Before returning, all chunks are deduplicated by SHA-1 ID so ChromaDB's upsert receives only unique chunks.

Q35
Why is a SHA-1 hash used as the chunk ID? Isn't SHA-1 broken?
MEDIUM›

The chunk ID formula is sha1(f"{file_path}:{start}:{end}:{content[:80]}"). SHA-1 is used here as a content-addressable key, not for cryptographic security. The goal is: same chunk content → same ID (for idempotent upserts). SHA-1 collisions exist but are astronomically rare for code content — two different code chunks producing the same SHA-1 hash essentially never happens in practice.

For cryptographic security (password hashing, tokens), SHA-1 is indeed broken. For content-addressable IDs (like Git uses SHA-1 for commit hashes), it's still widely acceptable. If needed, SHA-256 would be a drop-in improvement with no meaningful performance cost.

Q36
How does the GitHub URL normalization work?
MEDIUM›

User-provided GitHub URLs come in many forms. The parser uses regex to extract owner and repo from any format: re.match(r"^https?://(?:www\.)?github\.com/([^/]+)/([^/]+)", url). It then reconstructs the canonical git clone URL: https://github.com/{owner}/{repo}.git.

Branch detection: if the URL contains /tree/{branch}/path, it parses out the branch name and passes it to Repo.clone_from(url, target, depth=1, branch=branch). If the branch clone fails (branch doesn't exist), it falls back to cloning the default branch. depth=1 is a shallow clone — only the latest commit, no history — which is much faster for large repos.

Q37
Why does the parser filter out node_modules, dist, .venv and similar directories?
EASY›

The IGNORE_DIRS set excludes: .git, node_modules, dist, build, .venv, venv, __pycache__, target, .next.

Without this filter, node_modules alone can contain thousands of third-party files (hundreds of MB), making indexing take 10-100× longer and polluting the vector store with irrelevant library code. Users never ask about their dependencies' implementation details — they ask about their own code. The filter also enforces a 1MB file size limit to prevent indexing generated/minified bundles.

Q38
What is a DependencyEdge and how is the dependency graph constructed?
MEDIUM›

A DependencyEdge has: source (file path), target (imported module or symbol), kind (imports/calls/contains/api). Edges are created in three ways:

  1. Import edges: For each file, extract import statements (AST for Python, regex for others). Each import → DependencyEdge(source=file, target=imported_module, kind="imports").
  2. Contains edges: Each symbol in a file → DependencyEdge(source=file, target=symbol.name, kind="contains").
  3. The analysis service uses these edges to build the React Flow visualization graph — nodes are files/symbols, edges are dependencies.
Q39
How does the tempdir cleanup work and why is it important?
EASY›

The parser creates a temporary directory with tempfile.mkdtemp(prefix="codeinsight-"), clones the repository into it, processes all files, then in a finally block calls shutil.rmtree(workdir, ignore_errors=True).

The finally block is critical — it runs even if parsing raises an exception. Without it, every failed parse would leave cloned repositories on disk (potentially gigabytes). The ignore_errors=True flag prevents the cleanup itself from raising if the temp dir was partially created. This is the RAII (Resource Acquisition Is Initialization) pattern — acquire the temp dir at the start, guarantee release at the end.

Q40
How does the parser handle ZIP archives vs GitHub repositories?
EASY›

The _materialize_repository() function branches on request.sourceType:

  • "github": Normalizes the URL, constructs the git clone URL, runs Repo.clone_from(url, target, depth=1) (shallow clone for speed). Returns the cloned directory path.
  • "zip": Validates the archive path exists, extracts with zipfile.ZipFile.extractall(), checks if the ZIP contains a single top-level directory (common for GitHub-downloaded ZIPs), and returns that directory as the root. Otherwise returns the extraction root.

The single-directory detection handles the common case where a GitHub-downloaded ZIP extracts as repo-main/ with all files inside — the parser unwraps it automatically.

Q41
What languages does the parser support and how is language detection done?
EASY›

The SUPPORTED dict maps file extensions to language names: .py → Python, .ts/.tsx → TypeScript, .js/.jsx → JavaScript, .cpp/.cc/.hpp/.h → C++, .java → Java.

Language detection is purely extension-based: SUPPORTED.get(file_path.suffix.lower(), "Config"). Config files (package.json, Dockerfile, requirements.txt, etc.) are indexed as "Config" language for completeness. The language name is stored in each CodeChunk and used both for display and as metadata in the embedding text prefix.

Limitation: .h headers are assumed C++ (could be C). Binary files, images, and lock files are excluded by the 1MB size limit and extension filtering.

Q42
How does _guess_block_end work and what's the problem with it?
MEDIUM›

For non-Python languages using regex-based symbol detection, the parser can find where a function starts (via the regex match line) but can't reliably determine where it ends without a full parser. _guess_block_end(lines, start_line) simply returns min(len(lines), start_line + 80) — a fixed 80-line estimate.

Problem: This is a rough approximation. A 5-line utility function gets a 80-line chunk (includes following code). A 300-line class method is truncated at line 80. Both scenarios degrade embedding quality. The proper fix is using a real parser (tree-sitter) that provides accurate end_lineno for all languages, which is on the improvement roadmap.

CATEGORY 5 · 8 QUESTIONS
Streaming & SSE
Questions about Server-Sent Events, streaming LLM responses, proxy configuration, and client-side stream consumption.
Q43
What is SSE and how is it different from WebSockets?
EASY›

SSE (Server-Sent Events): One-directional, server → client push over a standard HTTP connection. The browser keeps an HTTP response open and receives data: ...\n\n events as they arrive. Built into browsers via EventSource API. Works over HTTP/1.1 and HTTP/2. Automatic reconnection on disconnect.

WebSockets: Bidirectional, full-duplex connection over a WebSocket handshake (HTTP upgrade). Both sides can send messages anytime. More complex, requires WebSocket server infrastructure.

For LLM token streaming, SSE is the right choice: the server sends tokens, the client never needs to send data mid-stream. SSE's simplicity, auto-reconnect, and HTTP compatibility make it ideal.

Q44
Why do you escape newlines in SSE tokens?
MEDIUM›

The SSE specification uses \n\n (double newline) as the event terminator. A single data: field ends at the next \n. If an LLM token contains a literal newline character (e.g., a code block line break), the data: field would terminate prematurely, and the browser would parse the remaining content as a new (malformed) event.

Fix: replace \n with the literal string \\n before yielding each token. The client reverses this: token.replaceAll("\\n", "\n"). This preserves the newline semantics in the displayed text while keeping SSE events well-formed.

Q45
How does FastAPI's StreamingResponse work?
MEDIUM›

StreamingResponse takes an async generator (or iterator) and streams its yielded chunks as the HTTP response body. It sets the Content-Type header from the media_type argument ("text/event-stream") and adds Transfer-Encoding: chunked to stream without a known Content-Length.

Each yield f"data: {token}\n\n" in the generator sends that chunk to the client immediately — FastAPI's ASGI server (uvicorn) flushes the TCP buffer after each yield. This is why the client sees tokens one-by-one rather than waiting for the full response.

Q46
What are the SSE custom events (citations, done) and how does the client handle them?
MEDIUM›

Standard SSE data events: data: token\n\n — received by eventSource.onmessage. Custom SSE events use the event: field: event: citations\ndata: [...]\n\n — received by eventSource.addEventListener('citations', handler).

The client sets up: eventSource.addEventListener('citations', e => { const cites = JSON.parse(e.data); appendCitationsToMessage(cites); }) and eventSource.addEventListener('done', () => eventSource.close()). This cleanly separates token content from metadata: the stream sends tokens first, then citations, then signals completion.

Q47
How does httpx handle streaming LLM API responses in Python?
MEDIUM›

httpx.AsyncClient.stream("POST", url, json=payload) opens an async HTTP connection with streaming enabled. The async with client.stream(...): context manager keeps the connection open. async for line in response.aiter_lines() yields each line of the response body as it arrives from the TCP stream.

For OpenAI-compatible APIs, each line is either data: {"choices":[{"delta":{"content":"token"}}]} or data: [DONE]. The code parses the JSON, extracts the token from choices[0]["delta"]["content"], and yields it. This creates a chain: LLM API → httpx line iterator → Python generator → FastAPI StreamingResponse → browser.

Q48
Why does X-Accel-Buffering matter for SSE proxies?
HARD›

Nginx (and other reverse proxies) buffer upstream responses by default — they accumulate bytes before forwarding to clients. For SSE streams, this means Nginx holds tokens until its buffer fills, then sends a batch — destroying the real-time streaming effect.

The X-Accel-Buffering: no header instructs Nginx to disable buffering for that specific response and forward each chunk immediately to the client. Without this header, deployed SSE streams behind Nginx show no tokens until the entire LLM response completes — the chat appears to hang, then all tokens appear at once.

Q49
How does the Node gateway proxy an SSE stream from the Python RAG service to the React client?
HARD›

The gateway sets response headers matching SSE requirements: Content-Type: text/event-stream, Cache-Control: no-cache, Connection: keep-alive. It then makes an HTTP GET/POST to the RAG service and pipes the response stream directly: pythonResponse.body.pipe(res) (Node streams).

The key is piping, not buffering. If the gateway did const body = await fetch(ragUrl).then(r => r.text()), it would wait for the entire response before forwarding — buffering. Piping connects the incoming stream to the outgoing stream byte-for-byte, maintaining the SSE timing semantics.

Q50
What happens if the SSE connection drops mid-stream?
MEDIUM›

Browser's native EventSource automatically reconnects after a configurable delay (default 3 seconds) if the connection drops. It sends a Last-Event-ID header with the ID of the last received event, allowing the server to resume from where it left off — but only if the server tracks event IDs and supports resumption.

In CodeInsight's current implementation, events don't have IDs and the server doesn't support resumption. A dropped SSE connection means the partial response is lost and the user must re-ask the question. For production, using id: fields in SSE events and server-side event journaling would enable resumable streams.

Auth & Security
JWT authentication, bcrypt password hashing, middleware patterns, and security considerations.
Q51
How does JWT authentication work in this project?
EASY›

Register: Client sends email + password. Server hashes password with bcrypt (10 rounds), saves user to MongoDB, returns jwt.sign({ sub: user._id }, JWT_SECRET, { expiresIn: '7d' }).

Login: Client sends credentials. Server looks up user by email, uses bcrypt.compare(password, user.passwordHash) to verify, returns a new JWT if valid.

Protected routes: Every request to a protected route must include Authorization: Bearer {token}. The requireAuth middleware extracts the token, calls jwt.verify(token, JWT_SECRET) — which validates the signature and expiry — and attaches req.userId = payload.sub for downstream use.

Q52
What is bcrypt and why use it over SHA-256 for passwords?
MEDIUM›

SHA-256 is a fast cryptographic hash — billions per second on modern GPUs. An attacker with a leaked database can try billions of password guesses per second (brute force / rainbow tables).

bcrypt is an adaptive password hashing function designed to be intentionally slow (work factor 10 = ~100ms per hash on typical hardware). Each hash includes a unique random salt, preventing rainbow table attacks. The cost factor is adjustable — as hardware gets faster, you increase the cost to keep up. bcrypt.compare() is constant-time, preventing timing attacks. These properties make bcrypt specifically suited for passwords.

Q53
What are the three parts of a JWT? Can you decode one without the secret?
MEDIUM›

JWT = base64url(header).base64url(payload).base64url(signature)

Header: {"alg":"HS256","typ":"JWT"} — the algorithm used to sign.

Payload: {"sub":"userId","iat":timestamp,"exp":timestamp} — the claims.

Signature: HMAC-SHA256(header + "." + payload, JWT_SECRET) — proves the server issued this token.

Can you decode without the secret? Yes — header and payload are just base64url-encoded (not encrypted). Anyone can decode them. But you cannot verify or forge a valid signature without the secret. The payload is visible but tamper-evident. For sensitive data, use JWE (JSON Web Encryption) instead of JWS (JSON Web Signature).

Q54
Why is the 7-day JWT expiry a security risk? What's the production solution?
HARD›

A 7-day JWT cannot be revoked before expiry. If a user's token is stolen (XSS, traffic interception), the attacker has 7-day access. There's no server-side session to invalidate.

Production solution — Refresh Tokens: Issue short-lived access JWTs (15 minutes) + long-lived opaque refresh tokens (30 days) stored in a database. The client uses the access JWT for API calls. When it expires, the client exchanges the refresh token for a new access JWT. To "logout" or revoke access, delete the refresh token from the database — the next exchange attempt fails. This gives the security of short-lived tokens without forcing re-login every 15 minutes.

Q55
How does the auth middleware prevent unauthorized access to other users' repositories?
MEDIUM›

Two layers: (1) Authentication: The requireAuth middleware verifies the JWT and attaches req.userId. Unauthenticated requests are rejected at this layer. (2) Authorization: Route handlers query MongoDB with both the repository ID from the URL and ownerId: req.userId: Repository.findOne({ _id: req.params.id, ownerId: req.userId }). If the repo exists but belongs to a different user, this query returns null → 404 (not 403, to avoid confirming the repo exists).

Without the ownership check, any authenticated user could access any repository by guessing or enumerating MongoDB ObjectIDs.

Q56
What is CORS and how is it handled in this project?
EASY›

CORS (Cross-Origin Resource Sharing) is a browser security mechanism that blocks JavaScript from making requests to a different origin (domain:port) than the page was loaded from.

In Express, cors() middleware sets response headers like Access-Control-Allow-Origin: http://localhost:5173, allowing the browser to make requests from the React dev server to the API. In production, this would be set to the actual frontend domain.

In development, the Vite proxy avoids CORS entirely by making the browser believe all requests are same-origin (localhost:5173). The proxy handles the actual cross-origin call server-side where CORS restrictions don't apply.

Q57
Where is the JWT token stored on the client and what are the security implications?
HARD›

The Redux slice stores the JWT in memory (JavaScript state) and typically in localStorage for persistence across page reloads.

localStorage security risk: Any JavaScript on the page can read localStorage — including injected scripts from XSS attacks. Storing JWTs there is a known security risk.

Recommended alternative: Store JWTs in httpOnly cookies. httpOnly cookies are inaccessible to JavaScript, making them immune to XSS-based token theft. The cookie is automatically sent with every request. The downside: CSRF attacks must be defended against (using SameSite=Strict cookies or CSRF tokens). For a developer tool, localStorage is acceptable; for a financial/sensitive app, httpOnly cookies are required.

Q58
What security risks exist with the ZIP file upload feature?
HARD›

Zip bombs: A maliciously crafted ZIP can expand from 1MB to 10GB+ (e.g., 42.zip). Without a decompressed size limit, this crashes the service. Fix: check total uncompressed size before extracting, reject if over a threshold (e.g., 500MB).

Path traversal: ZIP entries can have paths like ../../etc/passwd. Extracting naively could overwrite system files. Fix: validate that all extracted paths stay within the extraction directory.

Executable content: The extracted code is parsed but never executed — this is safe. But a future feature running code tests would introduce code execution risks requiring sandboxing.

File type checks: Verify the uploaded file is actually a ZIP (check magic bytes, not just file extension) before processing.

Q59
How would you implement rate limiting to protect against API abuse?
MEDIUM›

In Express: use express-rate-limit middleware. Apply globally for endpoint protection and more strictly for auth endpoints to prevent brute force: rateLimit({ windowMs: 15*60*1000, max: 100 }) (100 requests per 15 minutes).

For authenticated users, rate limit by req.userId (from JWT) instead of IP — prevents shared IP false positives. For the LLM chat endpoint specifically, limit by user ID and track token usage against a monthly quota.

Distributed rate limiting (multiple gateway replicas) requires a shared counter store — Redis with a sliding window algorithm (token bucket or leaky bucket) is the standard solution.

Q60
Why is the JWT_SECRET stored as an environment variable and not hardcoded?
EASY›

Hardcoding JWT_SECRET = "mysecret" means: (1) it's committed to Git history — anyone with repo access can forge tokens for any user forever. (2) It can't be rotated without a code change + deployment. (3) Different environments (dev/staging/prod) would use the same secret.

Environment variables keep secrets out of source code. In production, they're injected by the deployment infrastructure (Kubernetes secrets, AWS Secrets Manager, Docker secrets) — never committed to Git. The .gitignore excludes .env files. This follows the 12-factor app methodology: config separated from code.

CATEGORY 7 · 10 QUESTIONS
Databases (MongoDB & ChromaDB)
MongoDB document modeling, Mongoose schemas, ChromaDB vector operations, and database design decisions.
Q61
Why MongoDB instead of PostgreSQL for this project?
MEDIUM›

MongoDB's flexible document model is ideal for the analysis field in the Repository schema — it stores the entire RepositoryAnalysis object (files, chunks, symbols, dependencies) which has variable structure depending on the repository. In PostgreSQL, this would require either a JSONB column (losing type safety) or complex normalized tables with joins.

The chat messages array as embedded sub-documents is natural in MongoDB and would require a separate table with foreign keys in PostgreSQL. MongoDB's Mongoose ODM in the Node ecosystem is mature and well-supported.

Tradeoff: MongoDB has weaker ACID guarantees, no foreign key enforcement, and complex aggregations are less elegant than SQL JOINs. For this use case, the flexibility outweighs the relational advantages.

Q62
Explain the Mongoose schema design — why are Chat messages embedded vs referenced?
MEDIUM›

MongoDB offers two relationship patterns: embedding (store related data in the same document) and referencing (store an ID that references another collection).

Messages embedded in Chat: Chat and its messages are always accessed together — you never need a message without its chat context. Embedding means one query fetches the entire conversation. This is a "has-many" relationship where the "many" is bounded (chat history) and accessed atomically.

Users and Repos referenced: Chat references ownerId and repositoryId by ObjectId. These are separate entities with independent lifecycles — you access users and repos without their associated chats. References avoid duplication (user data in one document, not copied everywhere).

Q63
What does timestamps: true do in Mongoose schemas?
EASY›

timestamps: true is a Mongoose schema option that automatically adds and manages two fields: createdAt (set on document creation, never updated) and updatedAt (updated every time the document is saved/updated). Both are ISO 8601 Date objects.

Without timestamps: true, you'd need to manually set these fields in every create/update operation — error-prone and easy to forget. They're essential for auditing, debugging ("when was this repo created?"), and displaying human-readable dates in the UI.

Q64
How does ChromaDB's get_or_create_collection work?
EASY›

client.get_or_create_collection(name=f"repo_{repository_id}") is an idempotent operation: if a collection with that name exists, it returns it; if not, it creates a new one and returns it. No need to check existence first, no error if it already exists.

This means the same code works correctly for both the first-time indexing of a new repository and re-indexing an existing one (e.g., after code changes). The subsequent upsert() calls update existing vectors or add new ones as needed.

Q65
What metadata is stored alongside vectors in ChromaDB and why?
EASY›

Each vector in ChromaDB is stored with: filePath (the source file), language (Python/TypeScript/etc.), startLine, endLine, symbol (the function/class name if applicable).

This metadata is returned alongside the vector search results and used to: (1) Generate citations for the UI (filePath:startLine-endLine). (2) Build the embedding text prefix for query-time context. (3) Filter results (future feature — filter by language or file path). (4) Display in the analysis visualization. Metadata is stored as flat key-value pairs (strings/numbers) — not nested objects — because ChromaDB's metadata schema doesn't support nesting.

Q66
How would you handle MongoDB connection failures in the Node gateway?
MEDIUM›

Mongoose's connect() includes built-in reconnection logic: it automatically retries connections on failure with exponential backoff. Setting serverSelectionTimeoutMS controls how long to wait before failing.

For production robustness: (1) Listen to Mongoose connection events (mongoose.connection.on('error', ...)) and log them. (2) Add a health check endpoint that returns 503 if MongoDB is disconnected. (3) Consider circuit breaker pattern — if MongoDB is down for >N seconds, immediately reject requests rather than queuing them. (4) Use MongoDB Atlas with multi-region replicas for zero-downtime during maintenance.

Q67
What is the Mixed type in the Repository schema's analysis field?
MEDIUM›

Mongoose's Mixed type (also Schema.Types.Mixed or just Object) tells Mongoose "this field can be anything — don't apply schema validation to it." The analysis field stores the entire RepositoryAnalysis object from the Python parser, which has a complex nested structure (files, chunks, symbols, edges) that would be tedious to fully schema-ify in Mongoose.

Tradeoff: Mixed disables Mongoose's change detection for that field — you must call repo.markModified('analysis') after updating it, or Mongoose won't know to include it in the MongoDB update operation. It also loses type safety on the Node side — TypeScript just sees it as any.

Q68
How does MongoDB ObjectId differ from a UUID and why does MongoDB use it?
MEDIUM›

ObjectId is a 12-byte (24 hex char) MongoDB-specific ID: 4 bytes timestamp + 5 bytes machine ID + 3 bytes counter. This means ObjectIds are roughly sortable by creation time (first 4 bytes = Unix timestamp).

UUID (v4) is 16 bytes of random data — completely random, no inherent ordering.

Why MongoDB uses ObjectId: The timestamp encoding allows ObjectId.getTimestamp() to recover creation time without a separate field. The machine ID + counter ensures uniqueness across distributed nodes without coordination — MongoDB can generate IDs in clients without a server round-trip. For this project, ObjectIds serve as the repository ID used in ChromaDB collection names (repo_{id}).

Q69
What indexes should be added to the MongoDB collections for performance?
HARD›

User collection: { email: 1 } unique index — login queries find by email, must be fast and unique.

Repository collection: { ownerId: 1 } — repository list for a user. { ownerId: 1, status: 1 } compound — filter by status within a user's repos. { ownerId: 1, _id: 1 } — ownership verification queries.

Chat collection: { repositoryId: 1, ownerId: 1 } — load chats for a specific repository by a specific user.

Without these indexes, MongoDB performs full collection scans (O(N)) for every query. The User email index is defined inline in the Mongoose schema (unique: true implies an index). Others should be added explicitly as compound indexes for optimal query coverage.

Q70
How does ChromaDB persist data across container restarts?
MEDIUM›

In the Docker Compose setup, ChromaDB runs as its own container (chromadb/chroma image) with a volume mount: the container's /chroma/chroma directory is mounted to a host volume. ChromaDB's PersistentClient writes all vector data (HNSW graphs, metadata) to this directory as files on disk.

When the container restarts, the volume persists and ChromaDB reads the existing data back into memory. Without a volume mount, ChromaDB data would be lost on container restart (ephemeral container filesystem). The Python services connect via chromadb.HttpClient(host="chromadb", port=8000) — calling the ChromaDB HTTP server, not the filesystem directly.

Q84

How does the health check endpoint help with Docker Compose startup ordering?
MEDIUM›

Every FastAPI service exposes GET /health returning {"ok": true, "service": "..."}. In docker-compose.yml, services can declare healthcheck directives: test: ["CMD", "curl", "-f", "http://localhost:8101/health"] with interval: 30s, retries: 3.

Services can then declare depends_on: { repository-parser: { condition: service_healthy } }. Docker Compose won't start the gateway until the parser service passes its health check — preventing the race condition where the gateway tries to call the parser before it's ready to accept connections. Without health checks, you'd see "connection refused" errors at startup that mask real bugs.

What is the 12-factor app methodology and how does this project follow it?
HARD›

The 12-factor app is a methodology for building scalable, maintainable software-as-a-service apps. Key factors this project follows:

  • Codebase: One repo, multiple services (monorepo)
  • Dependencies: Explicitly declared (requirements.txt, package.json)
  • Config: Stored in environment variables (.env), never in code
  • Backing services: MongoDB, ChromaDB, LLM APIs are attached resources, easily swappable
  • Build/Release/Run: Docker images are the build artifact, run with Docker Compose
  • Processes: Services are stateless (any request can hit any instance)
  • Port binding: Each service exposes itself on a specific port
  • Concurrency: Scale by adding process copies
  • Disposability: Fast startup, graceful shutdown (uvicorn handles SIGTERM)
  • Logs: Treated as event streams (stdout)

Q91
Tell me about a technical challenge you faced in this project and how you solved it.
MEDIUM›

Situation: During testing with large repositories, the embedding pipeline crashed with DuplicateIDError from ChromaDB.

Task: I needed to understand why duplicate IDs were being generated and fix it without slowing down the pipeline.

Action: I traced the issue to the two-pass chunking strategy — symbol-level chunks and sliding window chunks could overlap on the same line ranges, generating identical SHA-1 IDs. I added a deduplication step: after generating all chunks, iterate through them tracking seen IDs in a Python set(), skip any chunk whose ID already appeared. This runs in O(n) time with O(n) memory.

Result: The pipeline became stable across all repository sizes. The fix also improved embedding quality slightly by removing redundant chunks that ChromaDB would have overwritten anyway.

Q96
What did you learn about working with external APIs and LLMs?
MEDIUM›

Key lessons: (1) Always have a fallback: API keys expire, rate limits hit, providers have outages. The offline mock mode means the system is always demonstrable. (2) Abstract the provider early: I initially hardcoded Groq calls. Refactoring to the Strategy Pattern (one interface, multiple implementations) was painful mid-project. Design the abstraction layer before writing provider-specific code. (3) Streaming is hard to proxy correctly: Every layer in the stack must not buffer streams. (4) Token costs add up: LLM APIs charge per token. Designing good chunking (not sending 100 chunks to the LLM when 8 suffice) matters for both quality and cost. (5) Prompt engineering is iterative: The first prompt I wrote hallucinated. Adding "ONLY use provided context" and "call out uncertainty" improved grounding dramatically.
