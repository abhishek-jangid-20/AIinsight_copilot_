CodeInsight AI › What is RAG?
CORE CONCEPT — AI
What is RAG?
Retrieval-Augmented Generation — the fundamental AI architecture powering CodeInsight's chat feature. Understanding this is essential for explaining the entire project.

🧠 The Core Problem RAG Solves

Large Language Models (LLMs) like GPT-4 or Llama-3 are trained on data up to a cutoff date. They have no knowledge of your private codebase. If you ask "How does authentication work in my project?", a plain LLM will hallucinate an answer.

RAG solves this by retrieving the relevant code chunks first, then feeding them as context to the LLM — so the model answers from your actual code, not from its training data.

🎯 One-line definition RAG = Retrieve relevant documents from a vector store → Augment the LLM prompt with those documents → Generate a grounded answer.

The RAG Pipeline in CodeInsight

USER INPUT
User types a question in the chat panel
e.g. "How does the repository parser extract symbols from Python files?"
EMBEDDING
Query → Dense Vector
The question is encoded by sentence-transformers (all-MiniLM-L6-v2) into a 384-dimensional float vector — a point in semantic space.
RETRIEVAL
Approximate Nearest Neighbor Search in ChromaDB
ChromaDB finds the top-8 code chunks whose embedding vectors are geometrically closest to the query vector. Similarity is measured by cosine distance — chunks that talk about "Python symbol extraction" cluster near the query.
AUGMENTATION
Build the Prompt with Context Injection
The _build_prompt() function injects the retrieved code chunks verbatim into the LLM prompt: "Answer using ONLY the retrieved code context. [1] services/repository-parser-service/main.py:198-206\n..."
GENERATION
LLM Streams the Answer Token-by-Token
Groq / Gemini / OpenRouter generates an answer grounded in the provided context. Tokens stream via SSE back to the browser.
CITATIONS
Citations Sent After Completion
After the last token, a citations SSE event sends the filePath, startLine, endLine for every chunk used — shown in the UI as source links.

📐 Why Vector Similarity Works for Code

Neural embeddings map semantically similar text to nearby points in high-dimensional space. The query "extract Python symbols" and the code chunk containing ast.walk(tree) for extracting class/function nodes will land close together because the model learned that AST-walking relates to symbol extraction during pre-training.

This is fundamentally different from keyword search (BM25/TFIDF) which would only find chunks that literally contain the words "Python" and "symbols". Vector search finds conceptually related chunks even if they use different terminology.

❌ Keyword Search (Old Way)

  • Matches exact words only
  • Misses synonyms / paraphrasing
  • "extract functions" won't find ast.FunctionDef
  • Fast but semantically shallow

✅ Vector Search (RAG Way)

  • Matches semantic meaning
  • Finds conceptually related code
  • "extract functions" WILL find ast.FunctionDef chunks
  • Slightly slower but deeply relevant
CORE CONCEPT — ML
Embeddings & Vector Space
How raw text becomes a mathematical point in space, and why that enables semantic search over your entire codebase.

📐 What is an Embedding?

An embedding is a fixed-size list of floating-point numbers (a vector) that represents the meaning of a piece of text. In this project, all-MiniLM-L6-v2 produces 384-dimensional vectors.

Think of it like GPS coordinates for meaning: the text "function to parse imports" and actual code that parses import statements will produce coordinates that are very close to each other in 384-dimensional space, even if they share no literal words.

The Embedding Pipeline in Code

Pythonservices/common/embeddings.py
# Step 1: Lazy-load model with LRU cache (loads only once in memory)
@lru_cache(maxsize=1)
def _load_sentence_transformer():
    model_name = os.getenv("EMBEDDING_MODEL", "all-MiniLM-L6-v2")
    return SentenceTransformer(model_name)

# Step 2: Encode text to vector
def embed_texts(texts: list[str]) -> list[list[float]]:
    model = _load_sentence_transformer()
    vectors = model.encode(texts, normalize_embeddings=True)
    return vectors.tolist()  # NumPy → Python list for JSON serialization

🎯 Why normalize_embeddings=True?

L2-normalizing vectors (dividing each by its magnitude) ensures all vectors lie on the surface of a unit hypersphere. This makes cosine similarity identical to dot-product similarity, which ChromaDB can compute faster using HNSW (Hierarchical Navigable Small World) graphs.

Without normalization, two conceptually similar but differently-worded texts might have very different magnitudes, throwing off similarity calculations.

🔄 The Fallback Hash Embedding

When the ML model can't load (offline development, no GPU/memory), _hash_embedding() creates a deterministic approximation: it tokenizes the text, SHA-256-hashes each token, maps hashes to vector positions, increments those positions, then L2-normalizes. It's not semantic, but it's consistent — the same text always produces the same vector, so retrieval still works (poorly).

How Vectors are Stored — The Embedding Text Format

A query like "Python function that parses imports" will retrieve chunks with language: Python and code about import parsing even closer together.

Microservices Architecture
Why CodeInsight is split into multiple independent services instead of one monolith, and what tradeoffs that creates.

🏗 Monolith vs Microservices

A monolith bundles everything — parsing, embedding, RAG, auth, UI — into one deployable unit. Simple to start, but as it grows, one slow operation (like loading a 400MB ML model) blocks everything else.

Microservices split the system into small, independently deployable services that communicate over HTTP/REST. Each service owns its domain, scales independently, and can be written in the best language for that job.

🔧 The Shared Common Module Pattern

All Python services import from services/common/. This directory contains embeddings.py, llm.py, vector_store.py, and models.py. In Docker, this directory is mounted into every service container and added to PYTHONPATH.

This is a deliberate design to avoid copy-paste across services. DRY (Don't Repeat Yourself) principle applied at the microservice level.

CORE CONCEPT — NETWORKING
Streaming & Server-Sent Events
How tokens flow from the LLM to the user's browser in real-time — the technology behind the ChatGPT-style typing effect.

⚡ Why Not Just Return the Full Response?

LLM responses can take 10-30 seconds for complex code questions. Without streaming, the user stares at a blank screen. With streaming, they see tokens appear immediately — perceived latency drops dramatically and the UX feels alive.

The alternative (WebSockets) requires a persistent bidirectional connection. SSE (Server-Sent Events) is simpler: one-way server-to-client push over a standard HTTP connection.

SSE Protocol Deep Dive

🔄 The Token Escape Problem

SSE uses \n\n (double newline) to delimit events. LLM tokens can contain literal newlines — which would prematurely terminate an event. The fix in rag-service/main.py:

escaped_token = token.replace("\n", "\\n")

The client then reverses this: content = content.replaceAll("\\n", "\n") to reconstruct proper newlines for display.

Python · FastAPIservices/rag-service/main.py
async def events():
    # Stream each LLM token as a separate SSE event
    async for token in stream_complete(prompt):
        escaped = token.replace("\n", "\\n")   # prevent event termination
        yield f"data: {escaped}\n\n"

    # Send citations after completion
    yield f"event: citations\ndata: {json.dumps(citations)}\n\n"
    yield "event: done\ndata: {}\n\n"

return StreamingResponse(events(), media_type="text/event-stream")

CORE CONCEPT — SECURITY
JWT Authentication
How stateless token-based authentication works, why it's preferred over sessions, and how it's implemented in the Express gateway.

🔐 Session vs Token Authentication

Session-based auth: Server stores a session ID in memory/database. Every request hits the DB to validate the session. Hard to scale horizontally (which server holds the session?).

JWT (JSON Web Token): The server signs a token with a secret key. Any server can verify the token by checking the signature — no DB lookup required. Stateless and horizontally scalable.

📦 JWT Structure

A JWT has 3 base64-encoded parts separated by dots: header.payload.signature

  • Header: Algorithm used (HS256) and token type (JWT)
  • Payload: Claims — { sub: userId, iat: timestamp, exp: timestamp }
  • Signature: HMAC-SHA256(header.payload, JWT_SECRET) — proves server issued it

The signature is what makes it tamper-proof. Change any bit of the payload, and the signature won't match.

TypeScriptserver/src/middleware/auth.ts
// Middleware: extract token from Authorization header, verify, attach user
export const requireAuth = (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1]; // "Bearer "
  if (!token) return res.status(401).json({ error: 'No token' });
  
  try {
    const payload = jwt.verify(token, process.env.JWT_SECRET);
    req.userId = payload.sub; // attach userId to request
    next();
  } catch {
    res.status(401).json({ error: 'Invalid token' });
  }
};
⚠️ Security Note — bcrypt rounds This project uses bcrypt.hash(password, 10). The cost factor (10) means the hashing takes ~100ms — slow enough to make brute-force impractical, fast enough for user registration. Higher values increase security but also increase latency.

🔵 What is ChromaDB?

ChromaDB is an AI-native vector database specifically designed to store high-dimensional embedding vectors and perform fast approximate nearest neighbor (ANN) search using HNSW (Hierarchical Navigable Small World) graphs.

When you call collection.query(query_embeddings=[...], n_results=8), ChromaDB traverses the HNSW graph to find the 8 vectors closest to your query vector in O(log N) time — far faster than brute-force L2 distance computation over all chunks.

📦 MongoDB — Document Model

MongoDB stores data as BSON documents (binary JSON). Its flexible schema is perfect for the analysis field in the Repository document, which stores the entire RepositoryAnalysis object (variable structure depending on the repo). Mongoose adds schema validation and TypeScript typing on top.

Key decision: repository analysis results are stored in MongoDB (not just ChromaDB) so the Node gateway can serve file trees, symbol lists, and dependency edges without querying the Python services again.

🔑 Isolation by Repository ID ChromaDB uses a separate collection per repository: repo_{repository_id}. This means you can delete a specific repository's vectors without affecting others, and semantic search is always scoped to the current repo.
CORE CONCEPT — RAG PIPELINE
Code Chunking Strategy
Why and how the repository parser splits files into chunks — the most critical decision affecting RAG retrieval quality.

✂ Why Chunk at All?

LLM context windows are limited (8K to 200K tokens). A codebase might have 500+ files, millions of tokens. You cannot embed an entire repo in one vector — that loses locality. You cannot include an entire file as context — that wastes the LLM's context window.

Chunking breaks files into semantically meaningful units that are small enough to embed precisely and large enough to contain useful context.

🔍
Two-Pass Chunking Strategy

Pass 1 — Symbol-Level Chunks: For each detected function/class, create a chunk covering exactly that symbol's line range. This ensures the embedding captures the full function context.

Pass 2 — Sliding Window Chunks: For lines NOT covered by symbols (config files, scripts, top-level code), create 120-line overlapping windows to ensure nothing is missed.

Pythonservices/repository-parser-service/main.py · _chunk_file()
def _chunk_file(repo_id, file_path, language, lines, imports, symbols):
    chunks = []
    
    # Pass 1: Symbol-level chunks (functions, classes)
    for symbol in sorted(symbols, key=lambda s: s.startLine):
        content = "\n".join(lines[symbol.startLine-1:symbol.endLine])
        if content.strip():
            chunks.append(_chunk(repo_id, file_path, language,
                                    symbol.startLine, symbol.endLine, content, imports, symbol))

    # Pass 2: Sliding window for uncovered lines
    window = 120
    for start in range(1, len(lines)+1, window):
        end = min(start + window - 1, len(lines))
        content = "\n".join(lines[start-1:end])
        if content.strip():
            chunks.append(_chunk(repo_id, file_path, language, start, end, content, imports, None))
    
    return chunks

🔑 Chunk ID Generation

Each chunk gets a deterministic SHA-1 ID: sha1(f"{file_path}:{start}:{end}:{content[:80]}"). This enables idempotent upserts to ChromaDB — if you re-index the same file, the same chunks produce the same IDs, so they overwrite rather than duplicate.

The deduplication step (seen_chunk_ids = set()) before calling ChromaDB prevents DuplicateIDError when the same symbol appears in multiple passes.

CORE CONCEPT — CODE ANALYSIS
AST Parsing Theory
How the repository parser understands code structure — the difference between regex-based and AST-based symbol extraction.

🌲 What is an AST?

An Abstract Syntax Tree (AST) is a tree representation of the syntactic structure of source code. Each node represents a construct (function definition, class, import statement, expression). The AST strips away irrelevant details (whitespace, comments) and exposes the code's grammar.

Python's built-in ast module can parse any valid Python source into an AST in microseconds. This is far more reliable than regex for extracting class/function names because it understands the language's grammar.

PythonAST Parsing vs Regex
# AST approach (reliable, grammar-aware):
tree = ast.parse(content)
for node in ast.walk(tree):
    if isinstance(node, ast.FunctionDef):
        # Exact line numbers, handles decorators, async, nested functions
        symbols.append(CodeSymbol(name=node.name, startLine=node.lineno, ...))

# Regex approach (fragile, used for non-Python languages):
pattern = re.compile(r"^\s*(?:export\s+)?(?:async\s+)?function\s+([A-Za-z_][\w]*)", re.MULTILINE)
# Fails on: inline functions, arrow functions, methods, overloaded functions

❌ Regex Limitations

  • Can't handle multi-line declarations
  • Fails on complex nesting
  • Language-specific — must write per-language regex
  • No accurate line range detection

✅ AST Advantages

  • Grammar-aware — handles any valid syntax
  • Precise lineno and end_lineno
  • Handles decorators, async, nested defs
  • Tree structure reveals containment
⚠️ Why Regex for JS/TS? Python's ast module only works for Python. For JavaScript/TypeScript, a proper parser (like Babel, acorn, or tree-sitter) would be ideal. This project uses regex as a pragmatic simplification — it catches the common patterns (class declarations, function declarations, Express routes) but misses arrow functions and complex patterns.

Prompt Engineering
How the RAG prompt is structured to maximize answer quality and prevent hallucinations.

✍ The RAG Prompt Template

Prompt engineering for RAG has specific challenges: you must ground the LLM in the provided context and prevent it from "leaking" its pretrained knowledge when the context is insufficient.

Pythonservices/rag-service/main.py · _build_prompt()
def _build_prompt(query: str, contexts: list[dict]) -> str:
    context_text = "\n\n".join(
        f"[{idx+1}] {item['metadata']['filePath']}:{item['metadata']['startLine']}"
        f"-{item['metadata']['endLine']}\n{item['content']}"
        for idx, item in enumerate(contexts)
    )
    return f"""Answer the developer's repository question using ONLY the retrieved code context.
Be specific, cite file paths and line ranges, and call out uncertainty.

Question:
{query}

Retrieved context:
{context_text}"""

CORE CONCEPT — ML MODELS
Sentence Transformers & CodeBERT
The ML models that power semantic search — how they work and why they're suited for code.

🔢 How Sentence Transformers Work

Sentence Transformers are BERT-based models fine-tuned on a contrastive learning objective: similar sentences are pulled together in embedding space, dissimilar ones are pushed apart. The result is an encoder that maps variable-length text to a fixed-size vector.

all-MiniLM-L6-v2 is a 6-layer MiniLM model distilled from a larger teacher. It's 80MB, runs on CPU, produces 384-dimensional embeddings in ~5ms per batch, making it practical for development without GPUs.

🧠

🎯 Why MiniLM Works for Code Search

Despite being trained primarily on natural language, MiniLM works surprisingly well for code because: (1) code comments and docstrings are natural language, (2) function/variable names are natural language tokens, and (3) the training includes technical text that shares vocabulary with code (GitHub READMEs, Stack Overflow).

CodeBERT is the correct choice for purely code-to-code search (e.g., "find functions similar to this snippet"), but for the natural-language-to-code use case in RAG, MiniLM's speed advantage makes it the better default.

Problems I Faced
Honest account of the challenges encountered during development — the kind of things that don't show up in tutorials but come up in every production project.

Problem 1: ChromaDB DuplicateIDError

🔴 The Problem

When the chunking pipeline ran on large repositories, the same code chunk was being generated multiple times — once from the symbol-level pass and once from the sliding window pass when they overlapped on the same line ranges. Sending duplicate IDs to ChromaDB's upsert() within the same batch caused a DuplicateIDError, crashing the entire embedding pipeline.

✅ The Fix

Added a deduplication step in the parser before returning chunks: iterate through all generated chunks, track seen IDs in a Python set(), and skip any chunk whose ID was already seen. The SHA-1 chunk ID (hash of path + line range + content prefix) acts as a content-addressable key that catches exact duplicates while allowing distinct overlapping windows.

💡 What I Learned

Always design for idempotency in data pipelines. Use content-addressable IDs (hash-based) so re-runs overwrite rather than duplicate. Test with repositories that have very short files to surface edge cases in chunking logic.

Problem 2: SSE Tokens Breaking on Newlines

🔴 The Problem

LLM responses for code contain literal newlines (in code blocks, bullet points). The SSE protocol uses \n\n as an event delimiter. When the LLM streamed a token containing \n, the browser's EventSource interpreted it as the end of an event, splitting the token stream into malformed events. Code blocks in the chat appeared garbled.

✅ The Fix

Escape newlines before yielding tokens: token.replace("\n", "\\n") in the Python SSE generator. On the client side, reverse the escape after receiving each token: rawToken.replaceAll("\\n", "\n"). This preserves newline semantics while keeping SSE events well-formed.

💡 What I Learned

Streaming protocols have specific format requirements that must be respected. Always test streaming with edge-case content (code blocks, JSON, markdown tables) not just simple prose. Read the protocol spec — SSE's double-newline delimiter is explicitly documented but easy to miss.

Problem 3: GitHub URL Normalization

🔴 The Problem

Users pasted GitHub URLs in many forms: https://github.com/user/repo, https://github.com/user/repo/tree/main/subfolder, https://github.com/user/repo.git, with or without trailing slashes. The direct Repo.clone_from(url) call failed on web URLs because they're not valid git clone URLs — they need https://github.com/user/repo.git.

✅ The Fix

Added URL normalization with a regex to extract owner/repo from any GitHub URL pattern, strip .git suffixes, reconstruct the canonical clone URL, parse out branch names from /tree/branch-name/ paths, and fall back to cloning the default branch if a specific branch clone fails.

💡 What I Learned

User input is always messier than you expect. Design for the happy path first, then enumerate edge cases systematically. For URLs specifically, always normalize before using — strip trailing slashes, handle different schemes (http vs https), and extract structured data with regex rather than string splits.

Problem 4: ML Model Memory Across Requests

🔴 The Problem

The initial implementation loaded SentenceTransformer(model_name) on every embedding request. Loading the 80MB model takes 3-5 seconds and consumes significant RAM. With concurrent requests, multiple model instances were loaded simultaneously, causing memory exhaustion in containers with 512MB RAM limits.

✅ The Fix

Wrapped the model loader with @lru_cache(maxsize=1) — Python's built-in memoization decorator. The first call loads the model; subsequent calls return the cached instance instantly. The maxsize=1 ensures only one model is ever in memory. Since lru_cache keys on function arguments, and the function takes no arguments (model name comes from os.getenv), the cache is effectively a singleton.

💡 What I Learned

ML models are heavyweight resources that must be managed with singleton patterns. @lru_cache is elegant for this in Python. In production, model loading should happen at service startup (warm-up), not on the first request — prevents the "cold start" latency spike.

Problem 5: Streaming Through the Node Gateway Proxy

🔴 The Problem

The Node.js gateway proxies the RAG service's SSE stream. Initially using http.get() and collecting the response body into a string before forwarding — this buffered the entire response, destroying the streaming effect. The user saw nothing until the full LLM response was complete.

✅ The Fix

Set the Node proxy to pipe the response stream directly: response.pipe(res) after setting the appropriate SSE headers (Content-Type: text/event-stream, Cache-Control: no-cache, X-Accel-Buffering: no). This passes each SSE chunk byte-for-byte to the client as it arrives from the Python service.

💡 What I Learned

HTTP proxies buffer by default. For streaming endpoints, explicitly disable buffering at every layer: the proxy, the client HTTP library, and even nginx/reverse proxies in production (via X-Accel-Buffering: no header). Think of streaming as a pipeline where any buffer in the middle destroys the streaming property.

Problem 6: Docker Networking Between Services

🔴 The Problem

Connecting to localhost:8101 from inside a Docker container refers to the container's own localhost, not the host machine or other containers. The Node gateway was configured with PARSER_URL=http://localhost:8101 and couldn't reach the Python parser service running in a different container.

✅ The Fix

Docker Compose creates a default bridge network. Services can reach each other using the service name as a hostname. Changed to PARSER_URL=http://repository-parser:8101 where repository-parser is the service name in docker-compose.yml. Docker's DNS resolver translates the service name to the container's IP automatically.

💡 What I Learned

Container networking is a different mental model from host networking. localhost inside a container is that container only. Service-to-service communication in Docker Compose uses service names as DNS hostnames. Always verify inter-service connectivity with a health check ping before integrating business logic.

🧠
RAG System Design Principles
  • Retrieval quality is more important than generation quality — a great LLM with bad chunks gives bad answers
  • Chunk size is a tradeoff: too small loses context, too large dilutes the semantic signal
  • Prepending metadata (file path, language, symbol name) to chunks significantly improves retrieval precision
  • Always normalize embeddings before storing — enables cosine similarity as dot product (faster ANN)
  • Per-repository collection isolation in ChromaDB enables clean deletion and prevents cross-repo retrieval contamination
🏗
Polyglot Microservices Architecture
  • Using the right language for each service is more important than having one language everywhere
  • A shared common library (mounted via Docker volumes + PYTHONPATH) eliminates code duplication across Python services
  • HTTP is a universal service contract — Node.js can call Python services without any shared types or generated clients
  • Health check endpoints (/health) on every service are essential for Docker Compose readiness probes
  • Service-name-based DNS in Docker Compose is simpler than hardcoded IPs and works across environments
⚡
Streaming APIs & Real-Time UX
  • SSE is the right choice for LLM token streaming — simpler than WebSockets, works over HTTP/1.1
  • Proxies must explicitly pipe streams, not buffer them — any buffering layer destroys streaming
  • Escape protocol-special characters (newlines in SSE) at the sender and unescape at the receiver
  • Post-stream events (citations, done signals) are elegant for metadata that arrives after content
  • Perceived latency is more important than actual latency — first token in 500ms beats full response in 3s
🔐
Security & Authentication
  • Never store plaintext passwords — bcrypt's adaptive hashing stays secure as CPUs get faster
  • JWT's statelessness makes it perfect for microservices — any service can verify without a central session store
  • Short token expiry + refresh token pattern is the production standard — 7 days is fine for development
  • CORS configuration must be explicit — allowing all origins is a security risk in production
  • All user-owned resources (repos, chats) need ownership checks — never trust client-provided IDs alone
🛠
Developer Experience & Tooling
  • TypeScript's type safety across the full stack catches integration bugs at compile time, not runtime
  • Pydantic models serve as both validation AND documentation for FastAPI endpoints
  • Docker Compose dramatically simplifies multi-service local development — one command starts 8 services
  • Vite's dev proxy eliminates CORS issues during development without changing any production config
  • Mock/offline fallbacks (LLM offline mode, hash embeddings) make the system runnable without any API keys
ENGINEERING JUDGMENT
Key Design Decisions
The intentional architectural choices made and the reasoning behind them — the kind of thing an interviewer will probe.

🎯 Decision 1: Node.js as API Gateway, NOT as an All-in-One Server

Why: Node.js excels at I/O-heavy work (auth, MongoDB, HTTP proxying) but is wrong for CPU-heavy ML operations. Python excels at ML (PyTorch, NumPy, transformers). Splitting into gateway + Python microservices uses each runtime for what it's best at.

Alternative considered: Python FastAPI for everything (no Node). Rejected because JWT+bcrypt+MongoDB tooling is better in the Node/TypeScript ecosystem.

🎯 Decision 2: ChromaDB Over PostgreSQL+pgvector

Why: ChromaDB is purpose-built for vector workloads with no configuration overhead. It handles embedding storage, metadata, and ANN search in a single API. pgvector requires a running PostgreSQL instance, schema migrations, and HNSW index configuration.

Tradeoff: ChromaDB is less battle-tested at scale than pgvector. For production at millions of chunks, pgvector or Pinecone/Weaviate would be more appropriate.

🎯 Decision 3: Repository-Scoped Collections in ChromaDB

Why: Each repository gets its own ChromaDB collection (repo_{id}). This enables clean deletion (delete collection = delete all vectors for that repo), prevents cross-repo retrieval contamination, and allows per-repository vector count queries.

Alternative: One global collection with a repository_id metadata filter. Simpler to manage but filtering is slower and deletion is harder (must query all IDs for that repo first).

🎯 Decision 4: Two-Pass Chunking (Symbol + Sliding Window)

Why: Symbol-level chunks (functions, classes) give the best semantic signal for named code entities. But files have top-level code, configs, and statements outside any symbol — the sliding window ensures 100% file coverage.

Tradeoff: Creates overlapping chunks between passes, requiring deduplication. The dedup cost is acceptable given the retrieval quality improvement from symbol-aware chunks.

🎯 Decision 5: Offline Fallback Mode

Why: Requiring API keys to run any part of the system makes development friction unbearable. The mock LLM mode (returns context verbatim) and hash embedding fallback allow the entire pipeline to run offline — useful for testing the parsing, chunking, and retrieval stages without any external dependencies.

Learning: Make systems runnable locally with zero external dependencies for development. Reserve production credentials for staging/production environments.

ROADMAP THINKING
Future Improvements
What would make this system better — shows forward-thinking and understanding of production engineering challenges.
🚀
Retrieval Quality Improvements
  • Hybrid search: Combine vector search (semantic) with BM25 (keyword) using Reciprocal Rank Fusion for better recall
  • Re-ranking: Add a cross-encoder re-ranker (e.g., ms-marco-MiniLM) to re-score the top-20 results and select the best 8
  • Tree-sitter parsers: Replace regex-based JS/TS/C++ parsing with tree-sitter for accurate ASTs in all languages
  • Parent document retrieval: Return parent chunks (larger context) when child chunks match — preserves function context
⚙
Production Architecture
  • Async queue: Move parsing/embedding to a background job queue (Celery + Redis or BullMQ) — remove blocking from the HTTP request cycle
  • WebSocket progress: Push real-time status updates instead of client polling for repository processing status
  • Rate limiting: Add request rate limiting per user to prevent abuse of LLM API budget
  • Caching: Cache frequent query embeddings (Redis) — same query on the same repo should skip re-embedding
  • Token counting: Track LLM token usage per user/repo for billing and quota management
🔐
Security Hardening
  • Refresh tokens: Implement short-lived access tokens (15min) + long-lived refresh tokens instead of 7-day JWTs
  • File upload validation: Scan ZIP archives for zip bombs and path traversal attacks before extraction
  • Secret management: Replace .env files with HashiCorp Vault or AWS Secrets Manager in production
  • Input sanitization: Sanitize repository names and chat messages before storing in MongoDB
